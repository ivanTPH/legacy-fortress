# Attachment And Document Architecture

This file describes the current attachment/document implementation in the repo today.

## Current shared architecture

## Canonical asset document flow

Primary helpers:

- [lib/assets/documentLinks.ts](/Users/ivan-imac/legacy-fortress-web/lib/assets/documentLinks.ts)

Core responsibilities:

- `resolveCanonicalAssetDocumentContext(...)`
  Resolves canonical asset ownership and wallet/org context.
- `createCanonicalAssetDocument(...)`
  Uploads a file to storage and inserts a row into `public.documents`.
- `getStoredFileSignedUrl(...)`
  Generates signed URLs for preview/download/print.
- `loadCanonicalDocumentWorkspaceData(...)`
  Loads canonical assets and their linked `documents` rows.

Storage behavior:

- bucket: `vault-docs`
- canonical storage path shape:
  - generated by `buildCanonicalDocumentStoragePath(...)`
  - uses section/category/asset context plus sanitized file name

## Shared preview UI

- [components/documents/AttachmentGallery.tsx](/Users/ivan-imac/legacy-fortress-web/components/documents/AttachmentGallery.tsx)

Current behavior:

- Image files:
  - thumbnail card
  - in-app image preview modal
- PDF / iframe-safe preview types:
  - in-app iframe preview modal
- Other file types:
  - file card
  - in-app modal with download fallback text

Live verified behavior:

- canonical bank/property/executor records show attachment summaries directly on saved cards
- legacy `SectionWorkspace` cards show attachment summaries directly on saved cards
- supported preview types:
  - images render in-app
  - PDFs render in-app
  - text files render in-app in an iframe preview
- non-previewable office-style files show the shared fallback message in-app and keep download available

This is now the shared preview surface that should replace page-level `window.open(...)` attachment UIs.

## Upload / render / remove / download logic

Upload:

- Canonical assets:
  - `createCanonicalAssetDocument(...)`
- Legacy section entries:
  - storage upload in `SectionWorkspace`
  - metadata persisted in `details.attachments[]`

Render:

- canonical records and documents workspaces use `AttachmentGallery`
- legacy section pages now also use `AttachmentGallery`

Remove:

- canonical records:
  - storage delete
  - row delete from `documents` or `attachments`
- legacy section entries:
  - storage delete
  - `details.attachments[]` update

Download:

- all current shared flows fetch signed/downloadable URLs and then download in browser
- preview fallback for non-previewable types keeps the download path visible in the same modal

## Confirmed mixed/older patterns still present

- `SectionWorkspace` still uses `section_entries`
- `public.attachments` still exists beside `public.documents`
- `app/(app)/vault/[section]/[id]/page.tsx` still bridges legacy `file_path`, but now routes preview/download through shared attachment UI instead of a one-off viewer
- legacy single-file schema fields still exist in migrations and table shapes (`file_path`)

## Rules for future prompts

- Use [AttachmentGallery.tsx](/Users/ivan-imac/legacy-fortress-web/components/documents/AttachmentGallery.tsx) for attachment lists and previews instead of creating a new page-local preview UI.
- Do not reintroduce `window.open(...)` preview flows where in-app preview is feasible.
- Prefer canonical `documents` plus [documentLinks.ts](/Users/ivan-imac/legacy-fortress-web/lib/assets/documentLinks.ts) for new document features instead of adding more `file_path` fields.
- If a legacy page still uses `SectionWorkspace`, extend its shared attachment path rather than building a second one-off uploader.
